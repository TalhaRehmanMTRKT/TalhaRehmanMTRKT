The previous post extended the electricity-only dispatch into a true multi-energy model: two CHPs, a heat-only boiler, a heat tank, and a district-heat trading link were added on top of the existing electric layer. Two balance equations (electric and heat) and a CHP heat-to-power coupling tied the two carriers together.

That model still treats every storage device as if it were always available. Real flexibility is rarely that polite: most of it sits in batteries that only connect for a few hours per day, the most visible example being plug-in electric vehicles (EVs). A fleet of EVs plugged into a microgrid is a time-windowed storage portfolio: each vehicle can charge or discharge only between its arrival and its departure, and it must leave with enough energy to be useful for its owner.

In this post we keep the entire model from the previous post unchanged and add an EV fleet on top of the electric bus.

Mathematical formulation

Decision variables

For every vehicle $n \in \{1,\dots,N_{\text{ev}}\}$ and every hour $t \in \{1,\dots,T\}$ we add three continuous variables:

  • $P_{\text{ev,chg}}^{n}(t),\, P_{\text{ev,dchg}}^{n}(t) \ge 0$: charging / discharging power of vehicle $n$.
  • $\text{evsoc}^{n}(t) \in [\text{SoC}_{\min}, \text{SoC}_{\max}]$: state of charge of vehicle $n$, with the same $\text{SoC}_{\min} = 0.1$ and $\text{SoC}_{\max} = 0.9$ bounds used by the battery and heat tank.
  • $u_{\text{ev}}^{n}(t) \in \{0,1\}$: per-vehicle charge / discharge selector.

The fleet is parameterised by per-vehicle arrival hour $t_a^{n}$, departure hour $t_d^{n}$, capacity $C_{\text{ev}}^{n}$, and initial SoC $\text{evsoc}_{\text{ini}}^{n}$.

Per-vehicle scheduling constraints

Three regimes are needed:

$$ \begin{aligned} & t = t_a^{n}: \; \text{evsoc}^{n}(t) = \text{evsoc}_{\text{ini}}^{n} + \frac{\eta_{\text{ev}} P_{\text{ev,chg}}^{n}(t) - P_{\text{ev,dchg}}^{n}(t)/\eta_{\text{ev}}}{C_{\text{ev}}^{n}}, \\ & t_a^{n} < t \le t_d^{n}: \; \text{evsoc}^{n}(t) = \text{evsoc}^{n}(t-1) + \frac{\eta_{\text{ev}} P_{\text{ev,chg}}^{n}(t) - P_{\text{ev,dchg}}^{n}(t)/\eta_{\text{ev}}}{C_{\text{ev}}^{n}}, \\ & \text{otherwise}: \; P_{\text{ev,chg}}^{n}(t) = P_{\text{ev,dchg}}^{n}(t) = 0. \end{aligned} $$

Departure-SoC constraint

To prevent the optimiser from draining the vehicle for cheap energy and returning it empty, we require it to leave at least half full:

$$ \text{evsoc}^{n}(t_d^{n}) \;\ge\; 0.5, \qquad n = 1,\dots,N_{\text{ev}}. $$

The charge / discharge powers are bounded so that the SoC stays within $[\text{SoC}_{\min}, \text{SoC}_{\max}] = [0.1, 0.9]$, exactly as for the stationary battery in the previous post:

$$ P_{\text{ev,chg}}^{n}(t) \le \frac{C_{\text{ev}}^{n}\,(\text{SoC}_{\max} - \text{evsoc}^{n}(t-1))}{\eta_{\text{ev}}}, \quad P_{\text{ev,dchg}}^{n}(t) \le C_{\text{ev}}^{n}\,(\text{evsoc}^{n}(t-1) - \text{SoC}_{\min})\,\eta_{\text{ev}}. $$

Mutual exclusion across all storage units

As in the previous posts, every storage device gets a Big-M binary selector that forbids charging and discharging in the same hour:

$$ B_{\text{chg}}(t) \le M\, u_{\text{bat}}(t), \quad B_{\text{dchg}}(t) \le M\,(1 - u_{\text{bat}}(t)), $$ $$ H_{\text{chg}}(t) \le M\, u_{\text{hst}}(t), \quad H_{\text{dchg}}(t) \le M\,(1 - u_{\text{hst}}(t)), $$ $$ P_{\text{ev,chg}}^{n}(t) \le M\, u_{\text{ev}}^{n}(t), \quad P_{\text{ev,dchg}}^{n}(t) \le M\,(1 - u_{\text{ev}}^{n}(t)), $$

with $M = 10^{6}$ (written 1e6 in the C++ code).

Modified power balance

The electric balance from the previous post gains a fleet-wide net term $\sum_n (P_{\text{ev,dchg}}^{n}(t) - P_{\text{ev,chg}}^{n}(t))$:

$$ \begin{aligned} & P_{\text{dg1}}(t) + P_{\text{dg2}}(t) + R_{\text{dg1}}(t) + P_{\text{chp1}}(t) + P_{\text{chp2}}(t) + B_{\text{dchg}}(t) - B_{\text{chg}}(t) \\ & \quad + \sum_{n=1}^{N_{\text{ev}}} \big( P_{\text{ev,dchg}}^{n}(t) - P_{\text{ev,chg}}^{n}(t) \big) + P_{\text{Gbuy}}(t) - P_{\text{Gsell}}(t) = P_{\text{load}}(t). \end{aligned} $$

The heat balance is unchanged.

Implementation in C++/CPLEX

The complete code lives in trk-blogs/p3/main.cpp and reuses every single line of the previous post that was not directly affected by the EV extension. The new pieces are listed below.

Fleet input data

int   numEvs   = 5;                        // Total number of EVs
float Eveffin  = 0.90f;                    // EV charge/discharge efficiency

// Per-vehicle arrival hour, departure hour, initial SoC and battery capacity.
// EV n is grid-connected for t in [ta[n], td[n]]; outside that window
// its charge and discharge powers are clamped to zero.
int*   ta       = new int[numEvs]   { 2, 4, 6, 9, 9 };
int*   td       = new int[numEvs]   { 6, 7, 9, 12, 16 };
float* evsocini = new float[numEvs] { 0.30f, 0.20f, 0.10f, 0.70f, 0.40f };
float* evcap    = new float[numEvs] { 38.3f, 47.5f, 28.9f, 56.0f, 52.0f };

Two-dimensional decision variables

CPLEX's IloArray<IloNumVarArray> typedef gives a clean way to expose vectors of vectors of variables, indexed first by vehicle and then by hour. A parallel IloArray<IloBoolVarArray> (BoolVar2D) holds one binary $u_{\text{ev}}^{n}(t)$ per vehicle and per hour.

IloBoolVarArray ubat(env, T);   // battery charge/discharge selector
IloBoolVarArray uhst(env, T);   // heat-storage charge/discharge selector
NumVar2D  Pevchg   (env, numEvs);
NumVar2D  Pevdischg(env, numEvs);
NumVar2D  evsoc    (env, numEvs);
BoolVar2D uev      (env, numEvs);
for (int n = 0; n < numEvs; n++)
{
    Pevchg[n]    = IloNumVarArray (env, T, 0, evcap[n], ILOFLOAT);
    Pevdischg[n] = IloNumVarArray (env, T, 0, evcap[n], ILOFLOAT);
    evsoc[n]     = IloNumVarArray (env, T, socmin, socmax, ILOFLOAT);
    uev[n]       = IloBoolVarArray(env, T);
}

Per-vehicle scheduling constraints

for (int t = 0; t < T; t++)
{
    for (int n = 0; n < numEvs; n++)
    {
        if (t == ta[n])
        {
            model.add(evsoc[n][t] == evsocini[n]
                      + (Eveffin * Pevchg[n][t] - Pevdischg[n][t] / Eveffin) / evcap[n]);
            model.add(Pevchg[n][t]    <= evcap[n] * (socmax - evsocini[n]) / Eveffin);
            model.add(Pevdischg[n][t] <= evcap[n] * (evsocini[n] - socmin) * Eveffin);
            // Big-M mutex: per-vehicle charge XOR discharge
            model.add(Pevchg[n][t]    <= 1e6 *      uev[n][t]);
            model.add(Pevdischg[n][t] <= 1e6 * (1 - uev[n][t]));
        }
        else if (t > ta[n] && t <= td[n])
        {
            model.add(evsoc[n][t] == evsoc[n][t - 1]
                      + (Eveffin * Pevchg[n][t] - Pevdischg[n][t] / Eveffin) / evcap[n]);
            model.add(Pevchg[n][t]    <= evcap[n] * (socmax - evsoc[n][t - 1]) / Eveffin);
            model.add(Pevdischg[n][t] <= evcap[n] * (evsoc[n][t - 1] - socmin) * Eveffin);
            model.add(Pevchg[n][t]    <= 1e6 *      uev[n][t]);
            model.add(Pevdischg[n][t] <= 1e6 * (1 - uev[n][t]));
            if (t == td[n]) model.add(evsoc[n][t] >= 0.5);
        }
        else
        {
            model.add(Pevchg[n][t]    == 0);
            model.add(Pevdischg[n][t] == 0);
        }
    }
}

Power balance with the EV fleet term

for (int t = 0; t < T; t++)
{
    IloExpr evCharge   (env);
    IloExpr evDischarge(env);
    for (int n = 0; n < numEvs; n++) {
        evCharge    += Pevchg[n][t];
        evDischarge += Pevdischg[n][t];
    }

    // Electric balance: EV charging is a load, EV discharging is a source.
    model.add(Pdg1[t] + Pdg2[t] + Rdg1[t]
              + Pchp1[t] + Pchp2[t]
              + Bdischg[t] - Bchg[t]
              + evDischarge - evCharge
              + PGbuy[t] - PGsell[t] == Pload[t]);

    // Heat balance is unchanged with respect to p2.
    model.add(Hhob[t] + Hchp1[t] + Hchp2[t]
              + HGbuy[t] - HGsell[t]
              - Hchg[t]  + Hdischg[t] == Hload[t]);
}

Results

Solving the program reveals the qualitative behaviour you would expect: EVs that arrive at low-price hours charge first; EVs whose connection window straddles a price peak get used for vehicle-to-grid discharge; and the at-departure constraint forces a final charging burst on vehicles whose SoC would otherwise have been drained too aggressively. Where to go next

A second user-side carrier we have ignored so far is cooling. Adding it requires not only a new balance equation but also two new conversion devices — an absorption chiller that turns heat into cooling and an electric chiller that turns electricity into cooling — which properly couples all three carriers into a single optimisation problem. We add it in the next post, which closes the series:
