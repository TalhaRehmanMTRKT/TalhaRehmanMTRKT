Most introductory optimisation tutorials stop at toy examples with two variables and a single constraint. They are useful to make sense of the abstract form of a linear program, but the gap between such a toy and the first real engineering problem you actually want to solve can feel wide. This post is the bridge I wish I had when I started: a complete, runnable C++/CPLEX program that solves a 24-hour economic dispatch of a small microgrid, with every modelling choice explained alongside the code.

The problem

Consider a microgrid that, over a horizon of $T = 24$ hours, must serve an electrical load $P_{\text{load}}(t)$ using four kinds of resources:

  • Two dispatchable generators (DG-1 and DG-2) with per-kW operating costs $C_{\text{dg1}}$ and $C_{\text{dg2}}$ and capacity limits.
  • Two non-dispatchable PV units with hourly forecasts $R_{\text{dg1}}(t)$ and $R_{\text{dg2}}(t)$.
  • A battery storage system characterised by capacity $P_{\text{bmax}}$, round-trip efficiency $\eta$, and an initial state of charge $\text{SoC}_{\text{ini}}$.
  • A connection to the upstream grid where energy can be bought at price $C_{\text{Gbuy}}(t)$ or sold back at $C_{\text{Gsell}}(t)$.

The goal is to choose, for every hour, how much each generator produces, how the battery is charged or discharged, and how much energy is exchanged with the grid, so as to minimise the total operating cost while meeting demand.

Mathematical formulation

Decision variables

For each hour $t \in \{1, \dots, T\}$:

  • $P_{\text{dg1}}(t),\, P_{\text{dg2}}(t) \ge 0$: dispatchable generator outputs.
  • $P_{\text{Gbuy}}(t),\, P_{\text{Gsell}}(t) \ge 0$: power bought from / sold to the grid.
  • $B_{\text{chg}}(t),\, B_{\text{dchg}}(t) \ge 0$: battery charging / discharging power.
  • $\text{SoC}(t) \in [0,1]$: battery state of charge.

Objective

Minimise the total operating cost over the horizon:

$$ \min \; \sum_{t=1}^{T} \Big[ C_{\text{dg1}}\, P_{\text{dg1}}(t) + C_{\text{dg2}}\, P_{\text{dg2}}(t) + C_{\text{Gbuy}}(t)\, P_{\text{Gbuy}}(t) - C_{\text{Gsell}}(t)\, P_{\text{Gsell}}(t) \Big]. $$

Constraints

Power balance. For every hour the supply must equal the demand:

$$ P_{\text{dg1}}(t) + P_{\text{dg2}}(t) + R_{\text{dg1}}(t) + R_{\text{dg2}}(t) + B_{\text{dchg}}(t) - B_{\text{chg}}(t) + P_{\text{Gbuy}}(t) - P_{\text{Gsell}}(t) = P_{\text{load}}(t). $$

Battery dynamics. The state of charge evolves as

$$ \text{SoC}(t) = \text{SoC}(t-1) + \frac{\eta\, B_{\text{chg}}(t) - B_{\text{dchg}}(t)/\eta}{P_{\text{bmax}}}, $$

with $\text{SoC}(0) = \text{SoC}_{\text{ini}}$, and the charge / discharge powers are bounded so that $\text{SoC}(t)$ stays inside $[0,1]$.

Implementation in C++/CPLEX

The complete code lives in trk-blogs/p1/main.cpp. We walk through it in six small pieces.

1. Input data

We hard-code the 24-hour profiles for load, prices, and PV forecasts in plain C arrays. A real workflow would read these from a CSV or a database; keeping them inline makes the example self-contained. 

int T     = 24;   // One day (hours)
int Cdg1  = 80;   // Cost per kW from DG-1
int Cdg2  = 90;   // Cost per kW from DG-2

int* Pload = new int[T] {
    169, 175, 179, 171, 181, 172, 270, 264,
    273, 281, 193, 158, 161, 162, 250, 260,
    267, 271, 284, 167, 128, 134, 144, 150 };

int* CGbuy = new int[T] {
     90,  90,  90,  90,  90,  90, 110, 110,
    110, 110, 110, 125, 125, 125, 125, 125,
    125, 125, 110, 110, 110, 110, 110, 110 };

int* CGsell = new int[T] {
     70,  70,  70,  70,  70,  70,  90,  90,
     90,  90,  90, 105, 105, 105, 105, 105,
    105, 105,  90,  90,  90,  90,  90,  90 };

float* Rdg1 = new float[T] {
    0,0,0,0,0,0,0,10,15,20,23,28,
    33,35,34,31,28,10,0,0,0,0,0,0 };

float* Rdg2 = new float[T] {
    0,0,0,0,0,0,0,10,15,20,23,28,
    33,35,34,31,28,10,0,0,0,0,0,0 };

float socini = 0.2f;   // Initial battery SoC
int   Pbmax  = 200;    // Battery capacity (kWh)
float effin  = 0.95f;  // Battery efficiency

2. Decision variables

CPLEX exposes IloNumVarArray for vectors of continuous variables. We pass a lower and upper bound directly at construction; this is equivalent to adding two box constraints and produces a slightly leaner model. 

IloNumVarArray PGbuy  (env, T, 0, IloInfinity); // Grid power bought
IloNumVarArray PGsell (env, T, 0, IloInfinity); // Grid power sold

IloNumVarArray statoc (env, T, 0, 1);   // Battery state of charge

IloNumVarArray Bchg   (env, T, 0, 100); // Battery charging
IloNumVarArray Bdischg(env, T, 0, 100); // Battery discharging

IloNumVarArray Pdg1   (env, T, 0, 80);  // DG-1 output
IloNumVarArray Pdg2   (env, T, 0, 100); // DG-2 output

3. Objective

The objective is built incrementally with an IloExpr. Selling to the grid appears with a negative sign because revenue offsets cost. 

IloExpr objective(env);
for (int t = 0; t < T; t++)
{
    objective += Cdg1 * Pdg1[t]
               + Cdg2 * Pdg2[t]
               + CGbuy[t]  * PGbuy[t]
               - CGsell[t] * PGsell[t];
}
model.add(IloMinimize(env, objective));

4. Constraints

Every constraint is added inside a single loop over $t$. The first hour is treated as a special case so the SoC dynamics can refer to $\text{SoC}_{\text{ini}}$ instead of $\text{SoC}(t-1)$. 

for (int t = 0; t < T; t++)
{
    // DG output bounds
    model.add(0 <= Pdg1[t]);  model.add(Pdg1[t] <= 80);
    model.add(0 <= Pdg2[t]);  model.add(Pdg2[t] <= 90);

    // Battery state of charge bounds
    model.add(0 <= statoc[t]); model.add(statoc[t] <= 1);

    if (t == 0)
    {
        model.add(statoc[t] == socini
                  + ((effin * Bchg[t] - (Bdischg[t] / effin)) / Pbmax));
        model.add(Bchg[t]    <= (Pbmax * (1 - socini) / effin));
        model.add(Bdischg[t] <= (Pbmax * socini * effin));
    }
    else
    {
        model.add(statoc[t] == statoc[t - 1]
                  + ((effin * Bchg[t] - (Bdischg[t] / effin)) / Pbmax));
        model.add(Bchg[t]    <= (Pbmax * (1 - statoc[t - 1])) / effin);
        model.add(Bdischg[t] <= Pbmax * statoc[t - 1] * effin);
    }

    // Power balance
    model.add(Pdg1[t] + Pdg2[t] + Rdg1[t] + Rdg2[t]
              + Bdischg[t] - Bchg[t]
              + PGbuy[t]   - PGsell[t] == Pload[t]);
}

5. Solving and reporting

IloCplex extracts the model, optionally exports the LP file for inspection, and solves it. We always check the return value of cplex.solve() before reading any value back: a model that is infeasible or unbounded will not have a meaningful objective. 

IloCplex cplex(env);
cplex.extract(model);
cplex.exportModel("ModelLP.lp");
cplex.setOut(env.getNullStream());

if (!cplex.solve()) {
    env.error() << "Failed" << endl;
    throw(-1);
}

double obj = cplex.getObjValue();
auto end   = chrono::high_resolution_clock::now();
auto ms    = chrono::duration_cast<chrono::milliseconds>(end - start);

cout << "\n\tElapsed time (ms): " << ms.count()  << endl;
cout << "Solution status     : " << cplex.getStatus() << endl;
cout << "Minimized objective : " << obj << endl;

6. Saving results to CSV

Finally we dump the optimal trajectories of every variable to output.csv so they can be plotted or post-processed. 

std::ofstream outputFile("output.csv");
if (outputFile.is_open()) {
    outputFile << "Time,Pload,CGbuy,CGsell,Rdg1,Rdg2,"
                  "PGbuy,PGsell,statoc,Bchg,Bdischg,Pdg1,Pdg2" << std::endl;

    for (int i = 0; i < T; i++) {
        outputFile << i + 1                 << ","
                   << Pload[i]              << ","
                   << CGbuy[i]              << ","
                   << CGsell[i]             << ","
                   << Rdg1[i]               << ","
                   << Rdg2[i]               << ","
                   <<  cplex.getValue(PGbuy[i])   << ","
                   << -cplex.getValue(PGsell[i])  << ","
                   <<  cplex.getValue(statoc[i])  << ","
                   << -cplex.getValue(Bchg[i])    << ","
                   <<  cplex.getValue(Bdischg[i]) << ","
                   <<  cplex.getValue(Pdg1[i])    << ","
                   <<  cplex.getValue(Pdg2[i])    << std::endl;
    }
    outputFile.close();
}

Results

Solving the program with the input data above yields a 24-hour dispatch plan that uses the cheapest generator (DG-1) at full capacity throughout the horizon, fills in the demand gap with DG-2 during the morning peak, charges the battery from the grid when prices are low, and sells back during the high-price midday window. Where to go next

This formulation is intentionally minimal so that every decision and constraint is easy to inspect. Natural extensions include adding binary on/off variables for the dispatchable generators (turning the LP into a MILP), modelling start-up costs and minimum up/down times, introducing reserve requirements, and accounting for forecast uncertainty through robust or stochastic formulations. We will pick these up in later posts of this series.
